# v1.4 "Dashboard" — Design

> Working design doc (drafted while the user is away, per `/loop`). Not code —
> decisions + open questions for when we start v1.4. Built on the agreed shape:
> **one tabbed window** (Activity / Cost / Usage); menu rows **deep-link** to the
> matching tab; the menu **slims back to a glance** as detail moves into the
> window. SwiftUI + Swift Charts, macOS 13+.

---

## 1. Goals & principles

- **The menu returns to a true glance.** Across v1.1–v1.3 the dropdown's "Today"
  section grew to ~6 lines. The dashboard is the "lean-in" surface; the menu goes
  back to at-a-glance. This is the *main justification* for the window — not just
  prettier charts.
- **One window, not many.** A single window with tabs. Every "click to see more"
  in the menu opens *this* window on the right tab. No per-feature windows.
- **Same chrome as Settings.** Reuse the existing NSWindow pattern
  (`AppDelegate.openSettings` → `NSWindow` + `NSHostingController(SwiftUI)` + the
  title-bar accessory). Consistent, low-risk.
- **Keep the aesthetic.** Monochrome / grayscale charts by default; the health
  dot stays the one colored element. (Open question on a single restrained accent.)
- **Don't tax the 60-second menu refresh.** The dashboard's heavier aggregates
  (per-model cost, daily series, 12–26-week heatmap) are computed **on demand**
  when the window opens / on a Refresh button — *not* on the always-running
  metrics tick.

### Non-goals (protect the glance identity)
- **Not** a full analytics suite or a replacement for `ccusage`/agentsview. Three
  focused tabs, not a wall of metrics.
- **Not** multi-tool (still Claude-only — our deliberate differentiator).
- **No** per-conversation / per-project drill-down in v1.4 (could be a later tab,
  but it pulls toward "dashboard sprawl" — out of scope unless the user asks).
- The window is the *opt-in* lean-in view; the **menu stays the product's center
  of gravity**. If the window ever feels load-bearing, we've drifted.

## 2. Architecture

### Window & navigation
- New `DashboardTab` enum: `.activity`, `.cost`, `.usage`.
- `AppDelegate.openDashboard(_ tab: DashboardTab)`:
  - Lazily create one shared `dashboardWindow` (mirror `settingsWindow`).
  - Set `dashboardModel.selectedTab = tab` (an `ObservableObject`).
  - `NSApp.activate` + `makeKeyAndOrderFront`.
- Root `DashboardView` observes `dashboardModel`; a **top segmented control**
  (`Picker(.segmented)`: Activity · Cost · Usage) switches tabs.
  - *Why segmented over a sidebar:* only 3 sections; segmented is compact and
    reads instantly. `NavigationSplitView` is overkill here (revisit if we add a
    Settings tab — see open Q5).
- Window: resizable (unlike Settings' fixed size — charts want room), sensible
  min size (~560×460), remember size/position via frame autosave.

### Deep-linking from the menu (the clickable-row mechanism already exists)
- We already have `ClickableMenuRowView` (built for the version link). Reuse it so
  the relevant menu rows are clickable → `openDashboard(tab)`:
  - **Activity strip / streak row** → `.activity`
  - **any cost line** (today $ / month / caching saved) → `.cost`
  - **5h / 7d / Sonnet rows** → `.usage`
- Subtle affordance: pointer cursor on hover (as the version row does); no blue
  highlight, to keep the menu calm.

### Data layer
A `DashboardData` provider (read on demand, off the 60s path):
| Tab | Source | Notes |
|---|---|---|
| Activity | jsonl daily token map | extend the existing `dailyTokens` window to ~12–26 weeks *for the window only* |
| Cost | jsonl per-model + per-day cost | new aggregates: cost-by-model map, daily-cost series (reuse `Pricing.swift`) |
| Usage | `HistoryStore` (5h/7d% over time) + live `UsageService.currentUsage` | history = charts; snapshot = current values/resets/pace |

Prep work this implies (design, not code yet):
- Extend `aggregateMetrics` (or a parallel on-demand aggregator) to also yield
  **per-model cost** and a **daily cost/token series** — we already iterate every
  entry, so it's additive.
- `HistoryStore` currently persists `h5`/`h7`. For a Sonnet history line we'd add
  `hSonnet` to `HistorySample` (schema bump → decode old files gracefully:
  optional field, default nil).

## 3. Per-tab content

### Activity tab
- **GitHub-style heatmap grid** — Swift Charts `RectangleMark`, x = week, y =
  weekday (Sun–Sat), opacity/grayscale = that day's token intensity. The full
  version of the v1.3.3 in-menu strip. Span: 12 weeks default (open Q4).
- **Streak stats** — current streak, longest, total active days, busiest day
  (reuse `currentStreak`/`longestStreak`; busiest = max daily).
- **Daily-token bars** — last 30 days, `BarMark`, under the heatmap.

### Cost tab
- **Headline** — today, this month, **projection** (reuse `monthlyProjection`).
- **Per-model breakdown** — Opus / Sonnet / Haiku → $ and %; small bar chart or
  table. (`Pricing.modelRate` already classifies models.)
- **Spend over time** — daily spend this month, `BarMark`/`LineMark`.
- **"Caching saved you $X"** callout, optionally a cumulative-savings line
  (uncached − actual, from `tokenCostUncachedUSD`).
- Persisting clarity: keep the **"at API rates"** framing prominent — this is
  API-equivalent value, not the subscription bill, not the prepaid balance.

### Usage tab
- **Current state** — 5h%, 7d%, Sonnet%, reset countdowns, the pace marker
  (live OAuth snapshot from `UsageService`).
- **Utilization history** — 5h and 7d `LineMark` over time from `HistoryStore`,
  showing the climb-and-reset sawtooth (the thing the in-menu sparkline only
  hints at). The dashboard is where the v1.3.4 store finally pays off.
- **Burn/pace context** — run-out ETA, on/ahead-of-pace.

## 4. Menu-slimming plan (the payoff)

Once the window holds the detail, collapse the menu's "Today" block from ~6 lines
to a 1–2 line glance, each clickable:
- e.g. **`Today: $X · streak Nd`** → clicks to Cost / Activity.
- Keep the 5h/7d rows (core glance) but make them clickable → Usage.
- Everything richer (per-model, charts, heatmap, history) lives in the window.

This is the deliberate "give the menu back its calm" move. *How aggressively* to
slim is the user's call (open Q3) — options range from "keep a one-liner + move
the rest" to "leave today's numbers, only move charts."

---

## 5. Intuitive-UX notes

- **Land on the right tab.** Deep-linking means a click never dumps you on a
  generic page — streak→Activity, cost→Cost, 5h→Usage. This is the single biggest
  intuitiveness win; make sure every clickable row maps obviously.
- **Refresh model.** Compute on open; a **Refresh** button + "updated HH:MM".
  Optional light auto-refresh only while the window is frontmost.
- **Empty states.** Fresh install has little history → each tab shows a friendly
  "collecting data…" rather than an empty chart (esp. Usage history & heatmap).
- **Consistency.** Same title accessory + close button as Settings; monochrome
  charts; restrained type. Possibly ⌘1/⌘2/⌘3 to switch tabs.
- **Discoverability.** Users should learn the rows are clickable — pointer cursor
  on hover + maybe a tiny chevron on the clickable glance rows.

## 6. Open questions for the user

1. **Tab UI:** top segmented control (recommended for 3) vs a sidebar?
2. **Color:** strictly monochrome/grayscale charts, or one restrained accent
   (e.g. heatmap intensity / "today" highlight)?
3. **Menu slimming:** how aggressive? (keep `Today: $X · streak Nd` one-liner and
   move the rest → my lean; alternative: keep today's numbers inline, move only
   charts.)
4. **Heatmap span:** 12 weeks (compact) vs 26/52 (more GitHub-like, more jsonl to
   read on open)?
5. **Settings as a 4th tab?** Fold the existing Settings window into the same
   window (Activity / Cost / Usage / **Settings**) for one unified window — or
   keep Settings separate? (Unifying is cleaner but a bigger refactor.)
6. **Window:** resizable + remember size/position — assume yes?
7. **Sonnet history:** worth bumping the `HistorySample` schema to also record
   Sonnet%, or keep history to 5h/7d for now?

### My recommended defaults (so you can just say "go with these")
1. **Segmented control** (3 tabs).
2. **Monochrome/grayscale** charts — consistent with the app; revisit color only
   if the heatmap reads poorly in grayscale.
3. **Slim aggressively:** menu keeps `Today: $X · streak Nd` + the 5h/7d rows
   (all clickable); everything else moves to the window.
4. **12-week heatmap** to start (cheap to widen later).
5. **Keep Settings separate** for v1.4 (fold-in is a nice v1.5 cleanup, not worth
   coupling the first dashboard build to a Settings refactor).
6. **Resizable + remembered frame:** yes.
7. **Add `hSonnet`** to the history schema now (cheap, optional-decode) so the
   Usage tab can show all three lines from day one.

### A note on initial data
The Usage-history charts and the heatmap only have data **from each feature's
install forward** (HistoryStore since v1.3.4; jsonl as far back as Claude Code
retains it, ~30 days typical). Early on the Usage tab will be sparse — the
"collecting data…" empty state matters most there.

## 7. Rough build order (when we start v1.4)

1. Window shell + tab switcher + `openDashboard(tab:)` + clickable menu rows
   (deep-link plumbing) — get the empty 3-tab window opening to the right tab.
2. **Usage tab** first — it has the cleanest data (HistoryStore + live snapshot)
   and proves the Swift Charts setup.
3. **Cost tab** — add per-model + daily-series aggregation, then charts.
4. **Activity tab** — the heatmap grid (most novel drawing) + streak stats.
5. **Slim the menu** + finalize deep-links + empty states.
6. README/ROADMAP, ship v1.4.x.

Each step is a normal feature-branch increment with the dev-install loop, same as
v1.1–v1.3.
